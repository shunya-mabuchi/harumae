# 非テキストファイル検査ロードマップ

AIまえチェックのファイル添付前チェックは、MVPではテキスト系ファイルだけを対象にします。PDF、docx、xlsx、画像OCRは本文の抽出方法、処理時間、ブラウザ権限、誤検出時の説明が重くなりやすいため、段階的に検討します。

この文書は、ユーザー本文やファイル本文を保存しない方針を維持したまま、どこまで検査できるかを整理するためのものです。

## 守る方針

- ファイル本文を永続保存しない
- ファイル本文を開発者のサーバーや外部LLM APIへ送信しない
- 抽出したテキストはブラウザ内のメモリ上だけで扱う
- placeholderMap、検出結果、送信履歴を永続保存しない
- `console.log`、エラー詳細、診断メモにファイル本文を含めない
- Chrome拡張の権限を増やす場合は、目的と影響をREADMEとストア説明に明記する
- 実装前に、対象サイトの添付UI、ファイルサイズ上限、処理時間、失敗時の表示を確認する

## 現在のMVP

対象:

- `.txt`
- `.md`
- `.csv`
- `.json`
- `.yaml` / `.yml`
- `.env`
- `.log`
- `.js` / `.ts`
- `.py`
- `.go`
- `.rb`
- `.java`
- `.html`
- `.xml`

処理:

1. `input[type=file]` で選択されたファイルを確認する
2. 対応するテキスト系ファイルだけ `File.text()` でブラウザ内読み取りする
3. `packages/core` のルールベース検出とrisk scoreを適用する
4. 高リスクまたは秘密情報保護対象があれば、安全化版ファイル作成またはキャンセルを促す
5. 読み取った本文とplaceholderMapは永続保存しない

対象外:

- PDF
- docx
- xlsx
- 画像OCR
- 対象サイト独自のドラッグ&ドロップ添付
- クリップボード経由のファイル添付

## PDF / docx / xlsx 検証結果

2026-06-24時点の判断は、0.1.xではPDF / docx / xlsxを解析しない、です。

理由:

- PDFはテキストレイヤーの有無、暗号化PDF、画像PDF、抽出順の崩れで結果説明が難しい
- docxはzip内XML、コメント、変更履歴、ヘッダー、フッター、脚注の扱いを決める必要がある
- xlsxはシート数、非表示シート、数式、セル書式、巨大ファイルの扱いを決める必要がある
- 追加ライブラリを入れると、バンドルサイズ、ライセンス、ブラウザ内完結、Web Worker対応の確認が必要になる
- 対応形式を増やすほど「安全に解析できた」と誤解されるリスクが増える

0.1.xでは追加ライブラリを採用しません。そのため、現時点でPDF / docx / xlsx向けの新しいライセンスリスクや通信リスクは増やしていません。

将来検討する場合の候補:

| 形式 | 方針 | 主な懸念 |
| --- | --- | --- |
| PDF | テキストレイヤーを持つPDFだけを候補にする | ライブラリサイズ、暗号化PDF、画像PDF、抽出順の崩れ |
| docx | zip内XMLから本文テキストだけを抽出する | ライブラリ依存、コメントや変更履歴の扱い、巨大ファイル |
| xlsx | セル文字列だけを抽出する | シート数、数式、非表示シート、巨大ファイル |

実装前チェック:

- 採用ライブラリのライセンス
- バンドルサイズ
- Web Worker対応
- ネットワーク通信の有無
- ブラウザ内処理だけで完結できること
- ファイル本文をログやエラーに含めない設計になっていること
- Chrome Web Store説明とプライバシーポリシーの更新要否

この段階でもOCRは入れません。文字抽出できないPDFや画像は、対象外として分かりやすく表示します。

## ドラッグ&ドロップ添付とクリップボード経由ファイル添付

2026-06-24時点の判断は、0.1.xでは動作保証の対象外、です。

理由:

- ChatGPT / Claude / Gemini / Perplexityの添付UIはサイトごとの実装差が大きい
- ドラッグ&ドロップは、ページ全体、composer周辺、非表示inputなど複数のイベント経路がある
- クリップボード経由ファイル添付は、ブラウザ、OS、サイト実装に依存しやすい
- 添付イベントを広く監視すると、意図しないファイルやサイトUIへ介入する可能性がある
- 0.1.xのChrome Web Store提出物では、必要最小限の権限と説明を優先する

将来対応する場合も、`<all_urls>` を無条件に要求しません。対象サイトと目的を限定し、SiteAdapterごとに添付イベントの入口を明文化します。

将来の調査観点:

- `input[type=file]` へ集約されるサイトか
- `drop` / `dragover` / `paste` のどのイベントでファイルが見えるか
- 添付前に安全化版ファイルへ置き換えられるか
- 置き換えられない場合、添付をキャンセルしてユーザーに案内できるか
- サイトの本来の添付UI、アクセシビリティ、ショートカットを壊さないか
- ファイル名、ファイル本文、現在URLをログや診断情報に含めないか

## ファイルサイズと処理時間の目安

実装前に次の上限を検証します。

- テキスト系ファイル: 1MBを初期目安にする
- PDF / docx / xlsx: 抽出後テキスト量で1MB相当を初期目安にする
- OCR: 1ページまたは1画像ずつ処理し、明示的な実行ボタンを置く
- 5秒以上かかる可能性がある処理は、進捗表示とキャンセル手段を必須にする

上限を超える場合は、本文を保存せずに「大きすぎるため解析できません」と表示し、ユーザーに手動確認を促します。

## UIで伝えること

- 対応しているファイル形式
- 対応外のファイル形式
- ファイル本文を保存しないこと
- 開発者サーバーや外部LLM APIへ送信しないこと
- 対応外ファイルは安全と判定したわけではないこと
- OCRや大きなファイルは時間がかかる可能性があること

対象外ファイルの文言案:

> この形式の本文解析にはまだ対応していません。ファイル本文は読み取りませんでした。添付前に内容を手動で確認してください。

## 現時点の結論

0.1.xではテキスト系ファイル添付前チェックを維持します。PDF / docx / xlsx / 画像OCRは、ロードマップ上の後続検討項目であり、現時点では解析しません。

対象外ファイルを「安全」と扱うのではなく、「AIまえチェックでは本文解析していないため、ユーザー自身で確認が必要」と明示します。

対象サイト独自のドラッグ&ドロップ添付やクリップボード経由ファイル添付も、0.1.xでは動作保証の対象外です。対応する場合はSiteAdapter単位で調査し、必要最小限の権限とUI説明を維持します。
