# 非テキストファイル検査ロードマップ

AIまえチェックのファイル添付前チェックは、MVPではテキスト系ファイルだけを対象にします。PDF、docx、xlsx、画像OCRは本文の抽出方法、処理時間、ブラウザ権限、誤検出時の説明が重くなりやすいため、段階的に検討します。

このロードマップは、ユーザー本文やファイル本文を保存しない方針を維持したまま、将来的にどこまで検査できるかを整理するためのものです。

## 守る方針

- ファイル本文を永続保存しない
- ファイル本文を開発者のサーバーや外部LLM APIへ送信しない
- 抽出したテキストはブラウザ内のメモリ上でのみ扱う
- placeholderMap、検出結果、送信履歴を永続保存しない
- `console.log`、エラー詳細、診断メモにファイル本文を含めない
- Chrome拡張の権限を増やす場合は、目的と影響をREADMEとストア説明に明記する
- 実装前に、対象サイトの添付UI、ファイルサイズ上限、処理時間、失敗時の表示を確認する

## 現在のMVP

対象:

- `.txt`
- `.md`
- `.csv`
- `.json`
- `.yaml` / `.yml`
- `.env`
- `.log`
- `.js` / `.ts`
- `.py`
- `.go`
- `.rb`
- `.java`
- `.html`
- `.xml`

処理:

1. `input[type=file]` で選択されたファイルを確認する
2. 対応するテキスト系ファイルだけ `File.text()` でブラウザ内読み取りする
3. `packages/core` のルールベース検出とrisk scoreを適用する
4. 高リスクまたは秘密情報保護対象がある場合は、安全版ファイル作成またはキャンセルを促す
5. 読み取った本文とplaceholderMapは永続保存しない

対象外:

- PDF
- docx
- xlsx
- 画像OCR
- 対象サイト独自のドラッグ&ドロップ添付
- クリップボード経由のファイル添付

## 初期拡張候補

最初に検討するのは、ブラウザ内で比較的安全にテキスト抽出できる形式です。

| 形式 | 方針 | 主な懸念 |
| --- | --- | --- |
| PDF | テキストレイヤーを持つPDFだけを候補にする | ライブラリサイズ、暗号化PDF、画像PDF、抽出順の崩れ |
| docx | zip内XMLから本文テキストだけを抽出する | ライブラリ依存、コメントや変更履歴の扱い、巨大ファイル |
| xlsx | セル文字列だけを抽出する | シート数、数式、非表示シート、巨大ファイル |

この段階ではOCRを入れません。文字抽出できないPDFや画像は、対象外として分かりやすく表示します。

## 後続拡張候補

OCRはブラウザ内で実行できますが、モデルやWASMの取得、CPU/GPU負荷、処理時間、対応言語、誤認識の説明が重くなります。そのため、別フェーズで検討します。

| 形式 | 方針 | 主な懸念 |
| --- | --- | --- |
| 画像OCR | ローカルOCRライブラリを候補にする | 初期ロード、精度、端末負荷、縦書きや日本語混在 |
| 画像PDF | PDFからページ画像化してOCRする | 処理時間、メモリ使用量、ユーザー体験 |
| スクリーンショット | 画像OCRと同じ扱いにする | 機密情報の誤認識、UI説明の難しさ |

OCRを入れる場合も、ファイル本文や画像を開発者サーバーへ送信しない方針を維持します。

## ファイルサイズと処理時間の目安

実装前に次の上限を検証します。

- テキスト系ファイル: 1MBを初期目安にする
- PDF / docx / xlsx: 抽出後テキスト量で1MB相当を初期目安にする
- OCR: 1ページまたは1画像ずつ処理し、明示的な実行ボタンを置く
- 5秒以上かかる可能性がある処理は、進捗表示とキャンセル手段を必須にする

上限を超える場合は、本文を保存せずに「大きすぎるため解析できません」と表示し、ユーザーに手動確認を促します。

## UIで伝えること

- 対応しているファイル形式
- 対応外のファイル形式
- ファイル本文は保存しないこと
- 開発者サーバーや外部LLM APIへ送信しないこと
- 対応外ファイルは安全と判定したわけではないこと
- OCRや大きなファイルは時間がかかる可能性があること

対応外ファイルの文言案:

> この形式の本文解析にはまだ対応していません。ファイル本文は読み取りませんでした。添付前に内容を手動で確認してください。

## 権限とリスク

現時点のMVPは対象サイトの `input[type=file]` に反応するだけで、追加の広い権限は要求しません。

将来的にドラッグ&ドロップ添付やクリップボード経由のファイル添付へ対応する場合は、サイト実装ごとのイベント監視が増える可能性があります。その場合でも、`<all_urls>` を無条件に要求せず、対象サイトと目的を限定します。

PDF / docx / xlsx / OCR対応で外部ライブラリを追加する場合は、次を確認します。

- ライブラリのライセンス
- バンドルサイズ
- Web Worker対応
- ネットワーク通信の有無
- ブラウザ内処理だけで完結できるか
- ファイル本文をログやエラーに含めない設計になっているか

## 現時点の結論

0.1.xではテキスト系ファイル添付前チェックを維持します。PDF / docx / xlsx / 画像OCRは、ロードマップ上の後続検討項目であり、現時点では解析しません。

対応外ファイルを「安全」と扱うのではなく、「AIまえチェックでは本文解析していないため、ユーザー自身で確認が必要」と明示します。
