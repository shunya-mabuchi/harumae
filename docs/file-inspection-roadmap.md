# 非テキストファイル検査ロードマップ

AIまえチェックのファイル添付前チェックは、MVPではテキスト系ファイルだけを対象にします。PDF、docx、xlsxは本文の抽出方法、処理時間、ブラウザ権限、誤検出時の説明が重くなりやすいため、段階的に検討します。画像OCRは外部API利用、サイズ、端末負荷、日本語精度の課題が大きいため、現時点では実装しない判断です。

この文書は、ユーザー本文やファイル本文を保存しない方針を維持したまま、どこまで検査できるかを整理するためのものです。

## 守る方針

- ファイル本文を永続保存しない
- ファイル本文を開発者のサーバーや外部LLM APIへ送信しない
- 抽出したテキストはブラウザ内のメモリ上だけで扱う
- placeholderMap、検出結果、送信履歴を永続保存しない
- `console.log`、エラー詳細、診断メモにファイル本文を含めない
- Chrome拡張の権限を増やす場合は、目的と影響をREADMEとストア説明に明記する
- 実装前に、対象サイトの添付UI、ファイルサイズ上限、処理時間、失敗時の表示を確認する

## 現在のMVP

対象:

- `.txt`
- `.md`
- `.csv`
- `.json`
- `.yaml` / `.yml`
- `.env`
- `.log`
- `.js` / `.ts`
- `.py`
- `.go`
- `.rb`
- `.java`
- `.html`
- `.xml`

処理:

1. `input[type=file]` で選択されたファイルを確認する
2. 対応するテキスト系ファイルだけ `File.text()` でブラウザ内読み取りする
3. `packages/core` のルールベース検出とrisk scoreを適用する
4. 高リスクまたは秘密情報保護対象があれば、安全化版ファイル作成またはキャンセルを促す
5. 読み取った本文とplaceholderMapは永続保存しない

対象外:

- PDF
- docx
- xlsx
- 画像、画像PDF、画像OCRが必要なファイル
- 対象サイト独自のドラッグ&ドロップ添付
- クリップボード経由のファイル添付

## PDF / docx / xlsx 検証結果

2026-06-27時点の判断は、0.1.xではPDF / docx / xlsxを解析しない、です。

理由:

- PDFはテキストレイヤーの有無、暗号化PDF、画像PDF、抽出順の崩れで結果説明が難しい
- docxはzip内XML、コメント、変更履歴、ヘッダー、フッター、脚注の扱いを決める必要がある
- xlsxはシート数、非表示シート、数式、セル書式、巨大ファイルの扱いを決める必要がある
- 追加ライブラリを入れると、バンドルサイズ、ライセンス、ブラウザ内完結、Web Worker対応の確認が必要になる
- 対応形式を増やすほど「安全に解析できた」と誤解されるリスクが増える

0.1.xでは追加ライブラリを採用しません。そのため、現時点でPDF / docx / xlsx向けの新しいライセンスリスクや通信リスクは増やしていません。

将来検討する場合の候補:

| 形式 | 方針 | 主な懸念 |
| --- | --- | --- |
| PDF | テキストレイヤーを持つPDFだけを候補にする | ライブラリサイズ、暗号化PDF、画像PDF、抽出順の崩れ |
| docx | zip内XMLから本文テキストだけを抽出する | ライブラリ依存、コメントや変更履歴の扱い、巨大ファイル |
| xlsx | セル文字列だけを抽出する | シート数、数式、非表示シート、巨大ファイル |

### 候補ライブラリの事前調査

2026-06-27にnpm registryの `npm view` で、代表的な候補ライブラリのライセンスとunpacked sizeを確認しました。ここでの数値はパッケージ単体の目安であり、実際のChrome拡張ZIPサイズ、依存関係、tree shaking、Worker分割後のサイズは別途PoCで測ります。

| 対象 | 候補ライブラリ | 確認したバージョン | ライセンス | unpacked size目安 | Web Worker方針 | 商用利用の見立て | 現時点の判断 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| PDF | `pdfjs-dist` | 6.0.227 | Apache-2.0 | 約35.6MB | pdf.js workerまたは専用Workerで抽出を分離する | ライセンス上は扱いやすいが、サイズとPDF差分が重い | 0.2系PoC候補。0.1.xには入れない |
| docx | `mammoth` | 1.12.0 | BSD-2-Clause | 約2.2MB | 専用WorkerでArrayBufferから本文抽出する | ライセンス上は扱いやすい | 0.2系PoC候補。コメント/変更履歴/ヘッダー等の扱いを先に決める |
| xlsx | `xlsx` | 0.18.5 | Apache-2.0 | 約7.5MB | 専用Workerでセル文字列抽出だけを行う | ライセンス上は扱いやすい | 0.2系PoC候補。非表示シート、数式、巨大ファイルを制限する |
| 画像OCR | `tesseract.js` | 7.0.0 | Apache-2.0 | 約1.4MB | OCR Workerが必要 | ライセンス上は扱いやすいが、言語データと端末負荷が重い | 現時点では採用しない |
| 日本語OCRデータ | `@tesseract.js-data/jpn` | 1.0.0 | MIT | 約18.2MB | OCR Workerへ読み込む必要がある | ライセンス上は扱いやすいが、追加ロードと保存領域が重い | 現時点では採用しない |

OCRはライブラリ本体だけでなく、日本語データや縦書きデータの取得・キャッシュ・端末負荷が問題になります。外部OCR APIは本文非送信とランニングコストゼロの方針に合わないため使いません。ブラウザ内OCRも、日本語精度、モデルサイズ、初回ロード、低スペック端末での待ち時間、誤検出時の説明が重いため、少なくとも0.2系の初期候補からは外します。

PDF / docx / xlsxを進める場合も、採用前に次を満たす小さなPoCを必須にします。

- 抽出処理はWeb Workerへ逃がし、UIスレッドを長く止めない
- 5秒以上かかる可能性がある場合は、進捗表示とキャンセルを必須にする
- 外部APIや開発者サーバーへファイル本文を送らない
- 対象外ファイル、暗号化ファイル、巨大ファイルを安全判定済みと表示しない
- Chrome Web Store提出ZIPのサイズ予算を悪化させない
- 抽出した本文、検出結果、placeholderMapを永続保存しない

実装前チェック:

- 採用ライブラリのライセンス
- バンドルサイズ
- Web Worker対応
- ネットワーク通信の有無
- ブラウザ内処理だけで完結できること
- ファイル本文をログやエラーに含めない設計になっていること
- Chrome Web Store説明とプライバシーポリシーの更新要否

この段階でもOCRは入れません。文字抽出できないPDFや画像は、対象外として分かりやすく表示します。

### 0.2系での判断基準

0.2系でPDF / docx / xlsxへ進む場合は、実装前に小さな検証PRで次を満たすことを条件にします。

| 観点 | 進める条件 | 進めない条件 |
| --- | --- | --- |
| ライセンス | 採用ライブラリが商用利用しやすく、再配布条件をREADMEへ説明できる | ライセンスやモデル/辞書データの再配布条件を説明できない |
| 通信 | ブラウザ内のFile/ArrayBuffer処理だけで完結する | 抽出処理に外部APIや開発者サーバーが必要になる |
| サイズ | Chrome Web Store提出ZIPのサイズ予算に収まる | 解析ライブラリでZIPや初期ロードが大きく悪化する |
| Worker | UIスレッドを長く止めず、キャンセルまたはタイムアウトできる | 大きなファイルでページ操作が固まる |
| 説明 | 「解析できない形式は安全判定ではない」とUIで説明できる | ユーザーに完全検査済みと誤解させる |

## 画像OCRの判断

2026-06-24時点の判断は、画像OCRは0.2系でも実装しない、です。

理由:

- 外部OCR APIを使うと、画像や抽出本文を外部サービスへ送る設計になりやすく、本文を外部へ送らない方針と相性が悪い
- 外部OCR APIは無料枠があっても、利用量、規約変更、課金、APIキー管理に依存し、永続的に無料のランニングコストゼロとは言いにくい
- ブラウザ内OCRは技術的には可能でも、日本語モデルや辞書データが重く、初回ロード、バンドルサイズ、端末負荷が大きくなりやすい
- 日本語OCRの精度が安定しない場合、検出漏れや誤検出の説明が難しく、ユーザーに「画像も検査済み」と誤解させるリスクがある
- Web Worker、進捗表示、キャンセル、ページ単位の処理、画像PDFとの切り分けなど、MVPの範囲を大きく超える

そのため、AIまえチェックでは画像や画像PDFを安全判定済みとは扱いません。UIとFAQでは「画像やOCRが必要なファイルは本文解析していないため、ユーザー自身で確認が必要」と説明します。

将来再検討する場合も、外部OCR APIではなく、ブラウザ内で完結する任意機能として、モデルサイズ、端末負荷、日本語精度、アクセシビリティ、Chrome Web Store説明を別Issueで再評価します。

## ドラッグ&ドロップ添付とクリップボード経由ファイル添付

2026-06-24時点の判断は、0.1.xでは動作保証の対象外、です。

理由:

- ChatGPT / Claude / Gemini / Perplexityの添付UIはサイトごとの実装差が大きい
- ドラッグ&ドロップは、ページ全体、composer周辺、非表示inputなど複数のイベント経路がある
- クリップボード経由ファイル添付は、ブラウザ、OS、サイト実装に依存しやすい
- 添付イベントを広く監視すると、意図しないファイルやサイトUIへ介入する可能性がある
- 0.1.xのChrome Web Store提出物では、必要最小限の権限と説明を優先する

将来対応する場合も、`<all_urls>` を無条件に要求しません。対象サイトと目的を限定し、SiteAdapterごとに添付イベントの入口を明文化します。

将来の調査観点:

- `input[type=file]` へ集約されるサイトか
- `drop` / `dragover` / `paste` のどのイベントでファイルが見えるか
- 添付前に安全化版ファイルへ置き換えられるか
- 置き換えられない場合、添付をキャンセルしてユーザーに案内できるか
- サイトの本来の添付UI、アクセシビリティ、ショートカットを壊さないか
- ファイル名、ファイル本文、現在URLをログや診断情報に含めないか

### SiteAdapter単位の添付経路検証表

対応可否はサイト単位で判断します。いずれのサイトでも、`<all_urls>` を無条件に追加しません。

| サイト | drop/dragover確認 | paste file確認 | 置き換え可能性 | 0.2系の初期判断 |
| --- | --- | --- | --- | --- |
| ChatGPT | composer周辺と非表示inputの関係を確認する | 画像/ファイルpaste時にDataTransferItemが見えるか確認する | 安全化版ファイルへ置換できなければキャンセル案内に留める | 調査対象 |
| Claude | ProseMirror周辺のdrop handlerと添付ボタンを確認する | clipboardData.filesの有無を確認する | サイトUIを壊さず置換できるか未確定 | 調査対象 |
| Gemini | rich-textarea周辺と添付ボタンのイベント経路を確認する | OS/ブラウザ差分を確認する | 送信本文とは別経路のため慎重に扱う | 調査対象 |
| Perplexity | 検索モード/添付ボタン周辺のDOM変更を確認する | ファイルpasteのサポート状況を確認する | 検索UIを誤操作しないことを優先する | 調査対象 |

調査結果が「安全に介入できない」の場合は、実装しない判断も正解とします。その場合は、FAQと対象外ファイルの文言を更新し、ユーザー自身で添付前確認が必要であることを明示します。

## ファイルサイズと処理時間の目安

実装前に次の上限を検証します。

- テキスト系ファイル: 1MBを初期目安にする
- PDF / docx / xlsx: 抽出後テキスト量で1MB相当を初期目安にする
- 5秒以上かかる可能性がある処理は、進捗表示とキャンセル手段を必須にする

上限を超える場合は、本文を保存せずに「大きすぎるため解析できません」と表示し、ユーザーに手動確認を促します。

## UIで伝えること

- 対応しているファイル形式
- 対応外のファイル形式
- ファイル本文を保存しないこと
- 開発者サーバーや外部LLM APIへ送信しないこと
- 対応外ファイルは安全と判定したわけではないこと
- 画像や画像PDFは本文解析しておらず、安全判定済みではないこと
- 大きなファイルは時間がかかる可能性があること

対象外ファイルの文言案:

> この形式の本文解析にはまだ対応していません。ファイル本文は読み取りませんでした。添付前に内容を手動で確認してください。

## 現時点の結論

0.1.xではテキスト系ファイル添付前チェックを維持します。PDF / docx / xlsxは後続検討項目です。画像OCRは外部OCR APIを使わず、ランニングコストゼロと本文非送信を維持する条件では負荷と精度の課題が大きいため、現時点では実装しません。

対象外ファイルを「安全」と扱うのではなく、「AIまえチェックでは本文解析していないため、ユーザー自身で確認が必要」と明示します。

対象サイト独自のドラッグ&ドロップ添付やクリップボード経由ファイル添付も、0.1.xでは動作保証の対象外です。対応する場合はSiteAdapter単位で調査し、必要最小限の権限とUI説明を維持します。
