# 安全化・マスク・秘密情報保護の整理

最終更新日: 2026-06-24

この文書は、AIまえチェックで使う「安全化」「マスク」「秘密情報保護」の意味を整理するためのものです。ユーザー向けの基本表現は「安全化」に寄せ、内部実装では必要に応じて `placeholder` / `mask` / `generalize` / `redact` などの方式名を使います。

## ユーザー向けの基本表現

### 安全化

ユーザーに見せる中心概念です。

AIサービスや外部フォームへ送る前に、検出された値や文章をそのまま残さず、より共有しやすい表現へ置き換えることを指します。

例:

- `taro@example.com` -> `[メールアドレス]`
- `090-1234-5678` -> `[電話番号]`
- `Project Blue Bridge` -> `[注意情報]`

UIでは原則として「安全化」「安全化対象」「安全化後」という表現を使います。

### マスク

安全化の一方式です。

値そのものを `[EMAIL_1]` や `[PHONE_1]` のような placeholder へ置き換える処理を指します。開発者向けAPIや内部型では、既存互換性のため `mask`、`maskedText`、`maskSensitiveText` という名前を残します。

ユーザー向けUIでは、独立した機能名として「マスク」を前面に出しすぎず、「安全化」の説明の中で必要な場合だけ使います。

### 秘密情報保護

処理方式ではなく、安全化が必須になるカテゴリまたはポリシーです。

APIキー、秘密鍵、JWT、`.env`、DATABASE_URL、Webhook URL、認証情報付きURLなどは、誤って外部へ送る影響が大きいため、「秘密情報保護の対象」として扱います。この対象が残っている場合、安全化なしでは送信できない方針にします。

## 内部方式

| 方式 | 役割 | 例 | 主な用途 |
| --- | --- | --- | --- |
| `placeholder` / `mask` | 検出範囲を連番 placeholder へ置換する | `[EMAIL_1]`, `[JWT_1]` | テスト、検出範囲の追跡、既存API互換 |
| `generalize` | カテゴリ別の日本語ラベルへ置換する | `[メールアドレス]`, `[電話番号]`, `[秘密情報]` | ユーザー向けの安全化結果 |
| `redact` | 値を残さない表現へ置換する | `[削除済み]` | 値の種類よりも内容を残さないことを優先したい内部変換 |

現在のUIで使う安全化結果は、基本的に `generalize` です。`placeholder` は内部API上の正式名で、既存互換のため `mask` も同じ方式のaliasとして残します。`redact` は `packages/core` のUI非依存な変換モードとして利用できますが、現時点の拡張UIではユーザーに独立した選択肢として表示していません。

## Finding / Ruleとの関係

現時点では、FindingやRuleに個別の推奨安全化方式を持たせていません。まずは riskLevel と DLPカテゴリから安全化結果を決めます。変換後の本文に具体値を残さない点は `generalize` / `placeholder` / `redact` で共通ですが、placeholderMapはメモリ上で元文字列を保持するため永続保存しません。

今後、より細かく制御する場合は、次のような方針を検討します。

- APIキー、秘密鍵、JWT、`.env` などは `placeholder` または `redact`
- メールアドレス、電話番号、URL、金額は `generalize`
- 人名、会社名、顧客名、案件名、採用・契約・給与・法務文脈は `generalize`
- ユーザーが明示的に選んだ値だけを安全化対象にする

## UI文言の方針

使う表現:

- 安全化して入力
- 安全化して貼り付け
- 安全化して送信
- 安全化対象
- 安全化後プレビュー
- 秘密情報保護の対象

避ける表現:

- Secret Guard
- Generalize
- Mask / マスクを独立機能のように見せる表現
- 完全に安全
- 100%検出
- すべての情報漏洩を防ぐ

## ドキュメントとテストでの扱い

テスト名や内部API名では、既存の `mask` / `maskedText` / `maskSensitiveText` を残す場合があります。これは互換性と実装責務を表すためで、ユーザー向けの機能名ではありません。

公開文書、LP、Chrome Web Store向け文書、Options Page、拡張モーダルでは、「安全化」を中心に説明します。秘密情報保護は、処理方式ではなく「安全化が必須になる対象」として説明します。
