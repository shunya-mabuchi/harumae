# AIまえチェック 脅威モデル

最終更新日: 2026-06-23

この文書は、AIまえチェックの設計上の守備範囲、信頼境界、想定する攻撃面、守れない範囲を明確にするための脅威モデルです。

AIまえチェックは、ChatGPT / Claude / Gemini などへ文章を貼り付ける前・送信する前に、個人情報・秘密情報・APIキーなどの消し忘れに気づくための補助ツールです。情報漏洩を完全に防ぐ製品ではなく、最終的に送信するかどうかはユーザーが判断します。

## Overview

AIまえチェックの主な実行面は次のとおりです。

- `apps/extension`: Chrome拡張本体。対象サイト上の貼り付け・送信前操作を検知し、確認モーダル、Options Page、site adapterを提供します。
- `packages/core`: ルールベース検出、マスキング、risk score、policy判定、リモートルール署名検証を担当します。
- `packages/llm`: WebLLMによる文脈リスク候補チェック、Worker、プロンプト生成、JSONパースを担当します。
- `apps/demo`: 紹介LP兼ミニデモ。拡張機能を入れる前に価値を試す補助体験です。
- `apps/worker`: Cloudflare Workers想定の署名付きルール配信APIです。ユーザー本文は受け取らず、署名付きルールJSONだけを返します。

守りたい主な資産は次のとおりです。

- AIサービスや外部フォームへ送る前の本文
- APIキー、秘密鍵、JWT、`.env` 形式の秘密情報、DATABASE_URL、Webhook URLなど
- メールアドレス、電話番号などの個人情報
- 顧客名、会社名、案件名、採用・契約・給与・法務などの文脈上注意したい情報
- 署名付きルール配信の改ざん耐性
- Chrome拡張の設定、検出ルールの有効/無効、WebLLM設定

明確に守れないもの、または保証しないものは次のとおりです。

- ユーザーが警告を確認したうえで送信した内容
- 検出ルールや文脈候補で拾えない未知の秘密情報
- 悪意ある別拡張、マルウェア、OSレベルの監視による読み取り
- ChatGPT / Claude / Gemini などのAIサービスへ送信された後の本文の扱い
- WebLLMによる完全な判定精度
- すべての情報漏洩の防止

## Threat Model, Trust Boundaries, and Assumptions

### 信頼境界

AIまえチェックでは、次の境界を意識します。

- ユーザーのブラウザ内と、開発者が管理するルール配信APIの境界
- 対象サイトのDOM / JavaScriptと、Chrome拡張のcontent script / extension pageの境界
- `packages/core` の確定的なルール検出と、`packages/llm` の補助的な文脈候補提示の境界
- 同梱ルールと、ネットワーク経由で取得する署名付きリモートルールの境界
- 一時的にメモリ上で扱う本文・検出結果と、`chrome.storage.local` に保存する設定の境界

### 前提

- ユーザー本文、検出結果、placeholderMap、送信履歴、AI文脈チェックの入出力本文は永続保存しません。
- `chrome.storage.local` に保存するのは、拡張機能の設定だけです。
- 開発者のサーバーや外部LLM APIへ、貼り付け本文・送信本文・添付前に読み取ったファイル本文を送信しません。
- WebLLMの初回利用時には、ローカル推論用のモデルファイルを取得する場合があります。これは本文送信ではありませんが、モデル配信元、ブラウザのキャッシュ、IndexedDBなどの保存領域に依存します。
- ルールベース検出は主判定として扱い、WebLLMの結果は「追加で確認したい候補」として扱います。
- 高リスク情報や秘密情報保護対象は、WebLLMの結果を待たずに安全側へ倒します。
- 署名付きルール配信が失敗した場合、ネットワークエラーの場合、署名検証に失敗した場合は、同梱ルールだけで動作します。
- 0.1.1では鍵ペアを再発行し、拡張側の公開鍵とWorker側の署名鍵を対応づけています。本番APIの署名検証は `pnpm qa:rules:production` で確認します。

### 入力の分類

攻撃者または外部環境の影響を受ける入力:

- ユーザーが貼り付ける本文、送信しようとする本文、添付前チェックで読み取るテキストファイル
- 対象サイトのDOM構造、入力欄、送信ボタン、ショートカット挙動
- WebLLMが返すJSON文字列や候補
- ネットワーク経由で取得するリモートルールJSON
- ブラウザ、WebGPU、WebLLMランタイム、モデル配信元の可用性

開発者が管理する入力:

- 同梱検出ルール
- 署名付き配信ルール
- ルール配信用の公開鍵・秘密鍵・`keyId`
- Chrome Web Store掲載文、README、プライバシーポリシー、サポートFAQ

ユーザーが管理する入力:

- 拡張機能の有効/無効
- 対象サイトごとのON/OFF
- 検出ルールごとのON/OFF
- WebLLMモデルIDとAI文脈チェック実行モード

## Attack Surface, Mitigations, and Attacker Stories

### 貼り付け・送信前本文

本文はもっとも重要な保護対象です。content scriptは対象サイト上の入力欄と送信前操作を扱うため、本文が一時的にメモリ上に乗ります。

主な対策:

- 本文、検出結果、placeholderMap、送信履歴を永続保存しない
- 本文を `console.log` やエラー詳細へ含めない
- 検出と安全化は `packages/core` のローカル処理を主役にする
- 高リスクまたは秘密情報保護対象は、安全化なしでは送信できない方針にする
- mediumリスクは詳細確認後にユーザーが判断できるUIにする

現実的な失敗シナリオ:

- 対象サイトのDOM変更により、送信前フックが効かない
- ルールにない形式の秘密情報を検出できない
- ユーザーが確認画面の内容を見落として送信する

### ルールベース検出とポリシー判定

メールアドレス、電話番号、JWT、APIキー風文字列、秘密鍵、`.env` などの確定的な情報は、WebLLMではなくルールベース検出で扱います。

主な対策:

- `packages/core` に検出ロジックを集約する
- riskLevel、カテゴリ、placeholder、maskingを型で扱う
- 重複範囲を優先順位で整理する
- 変換後テキストを再スキャンし、秘密情報が残っていないか確認する
- ルール追加時の観点を [detection-rule-authoring.md](./detection-rule-authoring.md) に残す

現実的な失敗シナリオ:

- 正規表現の過不足による検出漏れ・誤検出
- placeholder化や安全化後に、別の秘密情報が残る
- ルールカテゴリとUI文言のずれにより、ユーザーがリスクを理解しにくい

### WebLLM文脈チェック

WebLLMは、顧客名、人名、会社名、案件名、契約・採用・給与・法務など、ルールだけでは拾いにくい文脈上の注意候補を見つける補助機能です。

主な対策:

- WebLLMの結果を確定判定にしない
- 入力文に存在しない `surface` はFindingに変換しない
- confidenceが低い候補や長すぎる候補を除外する
- WebLLMが失敗しても、ルールベース検出と安全化は継続する
- エラー表示や診断メモに本文を含めない
- 実機互換性と復旧手順を [webllm-compatibility-matrix.md](./webllm-compatibility-matrix.md) と [webllm-error-recovery.md](./webllm-error-recovery.md) に分けて管理する

現実的な失敗シナリオ:

- WebGPU非対応、保存領域不足、モデル取得失敗によりWebLLMが使えない
- WebLLMがJSON形式を崩し、候補を読み取れない
- WebLLMが候補を見落とす、または不要な候補を返す

### 署名付きルール配信

ルール配信APIは、ユーザー本文を受け取らず、署名付きルールバンドルだけを返します。署名検証は、ネットワーク経由で取得したルールの改ざんや差し替えを検出するための仕組みです。

主な対策:

- リクエスト本文を使わず、`GET /api/rules/latest` だけにする
- 署名対象に `alg`、`keyId`、`payload` を含める
- 拡張側は公開鍵で署名検証できたルールだけを使う
- 署名検証失敗、HTTPエラー、URL未設定時は同梱ルールへフォールバックする
- 秘密鍵はリポジトリに置かず、Cloudflare Workersのsecretとして管理する
- 鍵ローテーションとロールバックを [rule-delivery-operations.md](./rule-delivery-operations.md) に残す

署名検証で守れること:

- 攻撃者がネットワーク上や配信経路でルールJSONを差し替えた場合に検出する
- 拡張が想定しない `keyId` や署名方式のルールを拒否する

署名検証だけでは守れないこと:

- 正規表現ルール自体の品質不足
- 開発者が誤ったルールを正規に署名して配信すること
- 秘密鍵が流出した状態で正しい鍵として署名されること

### 対象サイトadapter

ChatGPT / Claude / Gemini は、入力欄、送信ボタン、DOM構造、Reactイベントの扱いが異なります。

主な対策:

- 初期対象サイトを限定し、`<all_urls>` を無条件に要求しない
- site adapterごとの責務と手動QAを [extension-site-qa.md](./extension-site-qa.md) に残す
- 送信前操作だけでなく、貼り付け時にも確認できる
- 対象サイトのDOM変更時は、検出・送信前フック・入力反映のQAをやり直す

現実的な失敗シナリオ:

- 対象サイトのUI変更で送信ボタン検知が外れる
- 入力反映イベントが対象サイト側に伝わらない
- ページ側JavaScriptや別拡張が入力欄の内容にアクセスできる

### 設定保存

設定は `chrome.storage.local` に保存します。本文や検出結果は保存しません。

主な対策:

- 設定schemaと `settingsVersion` を持つ
- 認識しない設定キー、本文らしき値、壊れた値を正規化で落とす
- 保存前にも正規化する
- 設定マイグレーション方針を [options-settings.md](./options-settings.md) に残す

現実的な失敗シナリオ:

- 壊れた設定により、対象サイトやルールが意図せず無効化される
- 将来の設定追加時に、本文や検出結果を保存対象へ混ぜてしまう

## Severity Calibration (Critical, High, Medium, Low)

この重大度は、AIまえチェックの文脈で優先度を判断するための目安です。外部へ送信済みの本文の扱い、悪意あるOS/別拡張/マルウェアによる読み取り、ユーザーが明示的に警告を無視した送信は、原則として本リポジトリだけでは制御できない範囲です。

### Critical

該当例:

- 開発者のサーバー、外部LLM API、Analyticsなどへ、貼り付け本文・送信本文・添付前ファイル本文を送信してしまう
- 本文、検出結果、placeholderMap、送信履歴、AI文脈チェックの入出力本文を永続保存してしまう
- ルール配信用の本番秘密鍵や実APIキーをpublicリポジトリ、拡張ZIP、ログへ含めてしまう
- 高リスクまたは秘密情報保護対象を、安全化なしで自動送信できる実装にしてしまう

### High

該当例:

- host permissionsを不要に広げ、対象外サイトの入力欄まで読み取れる状態にする
- 署名検証に失敗したリモートルールを使ってしまう
- WebLLMの候補を確定判定として扱い、ユーザー確認なしで過剰または不適切な置換を行う
- 対象サイトadapterの不具合により、高リスク検出後でも送信抑止が効かない
- エラー詳細や診断メモに本文や検出文字列を含める

### Medium

該当例:

- 既知の秘密情報形式を検出できないルール品質の不足
- mediumリスクの説明が分かりにくく、ユーザーが判断しにくい
- WebLLM失敗時の復旧メッセージが不十分で、ルールベース検出が継続していることが伝わらない
- 対象サイトのDOM変更により、一部の貼り付け・送信前確認が効かない
- 壊れたリモートルール配信時のロールバック手順が不明確

### Low

該当例:

- README、LP、Chrome Web Store掲載文、Options Pageの文言が軽微にずれている
- WebLLM互換性マトリクスや手動QAメモの更新漏れ
- デモサイトの説明が、Chrome拡張本体との関係を十分に示せていない
- リスクラベルやplaceholder名の表記ゆれにより、UI理解が少し落ちる
